## Technical Notes
The Unreal Engine project was initialized as a C++ project to support custom destruction framework development. The project was integrated into the existing GitHub repository structure to ensure that source code, documentation and implementation evidence remain version controlled.

Chaos-related plugins were reviewed and enabled. Geometry Collection, Chaos Solver, Field System, Niagara and Chaos Niagara were selected as the initial destruction technology stack. The project was intentionally kept lightweight by avoiding unnecessary experimental systems.

The source architecture was divided into Core, Destruction, Networking and Utilities modules to establish separation of responsibilities before gameplay implementation begins.

Core framework classes were prepared to support future authority, synchronization and destruction workflows. Additional destruction-specific classes were created to isolate destruction management, local destruction behaviour and future replication logic.

The framework architecture prioritizes modularity and future multiplayer support rather than immediate gameplay functionality.

## Problems Encountered
- Git commit operations initially failed because Unreal-generated files inside Saved and cache directories were being tracked.
- Unreal source folders created manually were not immediately visible inside the editor until valid classes existed within them.
- Some Chaos plugin names differed from expectations and required verification inside the plugin manager.
- MVCDDestructionReplicationComponent exceeded Unreal naming limitations and required a shortened class name.
- Initial uncertainty existed regarding correct source folder placement inside the Unreal module structure.

## Solutions / Decisions
- Unreal-generated cache, temporary and Saved files were excluded from version control through Git ignore configuration.  
- Source architecture folders were moved inside the MVCD_Unreal module to follow Unreal module conventions.  
- Existing framework classes generated by the Unreal project template were retained instead of creating duplicate implementations.  
- The replication component was renamed to MVCDDestructionReplicationComp to comply with Unreal class naming limits.  
- Destruction architecture responsibilities were separated into dedicated framework classes rather than embedding destruction logic directly into gameplay actors.  
- The framework was designed with future multiplayer destruction support in mind.  
- Framework initialization was prioritized over gameplay implementation to establish a scalable foundation for later development.
